# Fase 5 — CI/CD: Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Criar um pipeline de CI no GitHub Actions que roda typecheck, lint, test e build em todo PR e push para `main`, protegendo o código limpo das fases anteriores de regressões futuras.

**Architecture:** Um único workflow `.github/workflows/ci.yml` com 4 steps encadeados (install → typecheck+lint → test → build), falhando no primeiro erro. Adiciona `packageManager` e `engines` ao `package.json` para fixar versões. Sem deploy automatizado (YAGNI), sem E2E, sem gate de cobertura.

**Tech Stack:** GitHub Actions, PNPM 10, Node 20 LTS.

## Global Constraints

- **PNPM 10.32.1** (versão em uso no repositório), lockfile v9.0.
- **Node 20 LTS** no CI (`ubuntu-latest`). Não usar Node 25 (versão local do dev) no CI — LTS é mais estável e reproduzível.
- `--frozen-lockfile` no install: falha se o lockfile não bate com `package.json` (pilha consistente).
- `typecheck` e `test` rodam como steps **separados** — Vitest usa esbuild (ignora tipos), então só rodar `test` esconde erros de tipo. Isso foi confirmado na Fase 4.
- **Limpar `.next` antes do typecheck** — o cache stale do Next causou falso-erro de typecheck na Fase 2 (tipos de rotas deletadas persistiam). Prevenir.

### YAGNI deliberado (não implementar)
- Deploy automatizado para Cloudflare no CI.
- E2E (Playwright) no CI.
- Husky / pre-commit hooks.
- Gate de cobertura (Codecov).

## Task 2: Criar o workflow `.github/workflows/ci.yml`

**Files:**
- Create: `.github/workflows/ci.yml`

**Interfaces:**
- Consumes: scripts `typecheck`, `lint`, `test`, `build` do `package.json`.
- Produces: pipeline que roda em PRs e push para `main`.

- [ ] **Step 1: Criar `.github/workflows/ci.yml`**

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

# Cancel in-progress runs for the same ref when a new commit is pushed.
concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  verify:
    name: Lint, Typecheck, Test & Build
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup PNPM
        uses: pnpm/action-setup@v4
        with:
          version: 10.32.1

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      # Next.js leaves stale generated types in .next/ that can cause false
      # typecheck errors after routes are deleted. Clear before typechecking.
      - name: Clean Next.js cache
        run: rm -rf .next

      - name: Typecheck
        run: pnpm typecheck

      - name: Lint
        run: pnpm lint

      - name: Test
        run: pnpm test

      - name: Build
        run: pnpm build
```

Notas de design (documentadas no workflow):
- `concurrency` cancela runs antigos do mesmo ref — economiza minutos quando você pusha vários commits seguidos num PR.
- `pnpm/action-setup@v4` antes de `actions/setup-node@v4` com `cache: pnpm` — habilita cache da store automaticamente.
- `rm -rf .next` antes do typecheck — previne o falso-erro visto na Fase 2.
- Steps separados (typecheck / lint / test / build) — falha no primeiro erro, e cada um fica visível isoladamente no log do Actions.
- `timeout-minutes: 15` — proteção contra hangs.

- [ ] **Step 2: Validar a sintaxe YAML**

Run: `node -e "const y=require('fs').readFileSync('.github/workflows/ci.yml','utf8'); console.log('YAML read ok, '+y.length+' bytes')"`
Expected: imprime o tamanho (confirma que o arquivo foi escrito). (Validação completa de schema do Actions acontece no runtime do GitHub.)

## Task 3: Validar o workflow localmente (simulação dos steps)

**Files:**
- Verify: nenhum (execução local dos mesmos comandos do workflow)

> GitHub Actions não roda localmente sem ferramentas extras (act). Esta task simula cada step do workflow na máquina local para confirmar que todos passam — provando que o workflow, quando rodar no CI, terá sucesso.

- [ ] **Step 1: Simular a sequência completa do workflow**

Run:
```bash
rm -rf .next && \
pnpm install --frozen-lockfile && \
pnpm typecheck && \
pnpm lint && \
pnpm test && \
pnpm build
```
Expected: todos passam sem erro (já validado nas fases anteriores, mas esta é a sequência exata do CI). Se qualquer step falhar, o workflow falhará no CI também — corrigir antes de seguir.

- [ ] **Step 2: Validar que `--frozen-lockfile` funciona**

O Step 1 já inclui `pnpm install --frozen-lockfile`. Se o lockfile estiver dessincronizado com o `package.json` (ex.: o `packageManager` recém-adicionado), este comando falha. Confirmar que passou — se falhou, rodar `pnpm install` (sem flag) para regenerar o lockfile, commitar, e re-rodar.

- [ ] **Step 3: Confirmar estado final**

Run: `git status --short`
Expected: working tree limpo (todos os commits feitos nas Tasks 1-2).

> Não há commit nesta task — é validação local. A validação real do workflow acontece no primeiro PR/push no GitHub.

## Notas

- O workflow **não inclui deploy**. Deploy para Cloudflare (`pnpm deploy`) continua manual. Automatizar exigiria secrets do Cloudflare no GitHub Actions (`CLOUDFLARE_API_TOKEN`, etc.) e cuidado com o `wrangler deploy` em CI — adiciona risco sem necessidade imediata.
- O cancel-in-progress (`concurrency`) economiza minutos de Actions em pushes consecutivos.
- Esta fase cria a base. Quando o projeto crescer, é trivial adicionar: deploy automatizado, E2E, cobertura, ou jobs paralelos (lint+typecheck paralelo a test+build).
- A Task 3 (validação local) é o melhor substituto viável para rodar o workflow antes de ele existir no GitHub. O verdadeiro teste de fogo é o primeiro PR.
